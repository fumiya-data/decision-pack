# 非機能要件

## 1. 文書の目的

この文書は、機能そのものではなく、実装・運用・変更・監査・性能・記録の前提条件を定義する。
Decision Pack は長期的に拡張される前提のため、初期段階から非機能要件を明示しておく。

## 2. 変更容易性

### 2.1 モジュール性

- モジュール境界が明確であること
- 顧客分析、在庫管理、意思決定計算が独立して変更できること
- `decision-engine` は一覧検索や顧客個票処理に依存しないこと
- GUI は API 契約を通してのみバックエンドへ依存すること

### 2.2 契約中心の変更

- 仕様変更は文書を先に更新すること
- DB 契約、API 契約、レポート JSON 契約の意味を明示してから実装すること
- 契約の互換性を壊す変更は、移行手順と影響範囲を伴うこと

## 3. 運用性

### 3.1 ジョブ追跡

- 各ジョブが `run_id` または `job_id` で追跡できること
- 実行開始、完了、失敗、再実行を区別できること

### 3.2 障害追跡

- 失敗時にログと失敗要因を辿れること
- どの入力、どのジョブ、どの時刻で失敗したかを追えること

### 3.3 再実行性

- ETL、分析、シミュレーションは再実行可能であること
- 再実行の結果と元実行を識別できること

## 4. 監査性

### 4.1 データの出所

- 原データ、変換後データ、分析結果、シミュレーション結果の出所が追跡できること
- 顧客データの整形失敗や補正結果を記録できること

### 4.2 意思決定根拠

- 在庫リスクや補充提案の元になった入力を説明できること
- 少なくとも、需要予測、在庫残高、補充制約、シナリオ条件を追跡できること

## 5. 性能

### 5.1 GUI

- GUI は重い計算を内包しない
- 一覧表示や詳細表示に必要な情報取得は API 越しに行う

### 5.2 バックエンド

- 重い処理は API の向こう側のジョブとして実行する
- 長時間処理は同期 API に閉じ込めない

### 5.3 将来拡張

- ローカル構成でも AWS 構成でも、同じ責務分割で動作できること

## 6. DB アクセス方針

### 6.1 基本方針

- ORM は採用しない
- DB 操作は原則として手書き SQL で明示的に行う
- schema の意味は migration SQL と仕様書で管理する

### 6.2 期待する効果

- DB の意味が ORM モデルへ吸収されにくいこと
- staging、集計、契約テーブルの意味を SQL で明示できること
- 変更履歴とレビュー対象が追いやすいこと

## 7. 命名と記録

### 7.1 文書

- 文書名と文書ディレクトリ名は英語 `kebab-case`
- 本文は日本語
- 進捗記録は `yyyy-mm-dd-hhmm-<summary>.md` 形式にする

### 7.2 意思決定

- 記録すべき判断は ADR に残す
- 後で設計や実装へ影響する判断は、口頭や会話だけで終わらせない

## 8. テストと検証

### 8.1 実装検証

- 主要な純粋ロジックは単体テスト可能であること
- 取込、分析、計算の境界は統合テスト可能であること

### 8.2 形式検証

- Lean 検証は、在庫計算や KPI 集約など意味が壊れやすい純粋ロジックを優先する
- すべてを Lean 化することは目的としない

## 9. 推薦高度化の移行条件

- 高度な推薦アルゴリズムへの移行は、データ量と履歴期間の条件を満たしたうえで判断する
- あわせて、MVP の業務指標または検証指標が基準を下回ることを確認してから移行判断する
- 高度化は「精度が上がりそうだから」ではなく、「MVP では不足することが確認できたから」行う

## 10. セキュリティと権限の扱い

- 本段階では厳密な権限管理の実装詳細はスコープ外とする
- ただし、GUI が DB や S3 を直接見ない構成は維持する
- API 境界で将来の認証・認可を追加できる余地を残す
