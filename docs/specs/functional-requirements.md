# 機能要件

## 1. 文書の目的

この文書は、Decision Pack が提供すべき機能をモジュール別に定義する。
ここに記載する機能要件は、画面、API、DB、ジョブ設計の基準となる。

## 2. 顧客データ前処理

### 2.1 入力

- 顧客マスタ相当の CSV
- 顧客を識別できる項目
- 顧客属性、連絡先、購入関連属性、任意メモを含みうる列

### 2.2 必須機能

- 顧客CSVを受け取り、正規化済み顧客データを生成できる
- 文字種、日付、数値、真偽値の揺れを吸収できる
- 列名の揺れや不要列を扱える
- 整形失敗、欠損、解釈不能値を issue として記録できる
- 正規化結果を PostgreSQL へ永続化できる
- 取込実行単位を `job_id` または `run_id` で追跡できる

### 2.3 出力

- `customers`
- `customer_load_issues`
- 必要に応じて `customer_segment_summary`

### 2.4 成立条件

- 同じ入力に対して、再実行しても同じ意味の結果になること
- 元データと正規化結果の対応関係を追跡できること

## 3. 商品・購入履歴・在庫

### 3.1 商品品目

- 商品品目リストを取り込める
- 品目を `item_id` で一意に扱える
- 品目名、カテゴリ、在庫関連属性を保持できる

### 3.2 購入履歴

- 受注と受注明細を取り込める
- 顧客が何を買ったかを `order` と `order_item` の組で追跡できる
- 顧客、注文、明細、商品が意味的に結びつくこと

### 3.3 在庫

- 在庫残高を品目単位で保持できる
- 必要に応じて在庫移動履歴を保持できる
- 在庫一覧の絞り込み検索ができる
- 品目カテゴリ別に在庫リスク判定へ必要な情報を参照できる

### 3.4 成立条件

- 商品、購入履歴、在庫が別々の CSV から入ってきても、一貫した `item_id` を軸に結びつくこと
- 顧客CSVだけでは購入履歴を作れない前提が保たれること

## 4. 購入傾向分析

### 4.1 必須機能

- 顧客ごとの購入履歴を集約できる
- 顧客ごとの次回購入候補 Top-N を算出できる
- 候補ごとに `item_id`, `score`, `rank`, `as_of` を保持できる
- 品目ごとの将来需要を集約できる
- `decision-engine` 向けに `item_id` 単位の需要予測を出力できる

### 4.2 MVP アルゴリズム

- 再購入確率を扱える
- 商品遷移確率を扱える
- セグメント人気を扱える
- MVP アルゴリズムは検証データを使って重み調整できる

### 4.3 非要求

- 初期段階では高度な機械学習モデルは必須としない
- 顧客個票を `decision-engine` へ直接渡すことはしない

## 5. 意思決定支援

### 5.1 入力

- 品目別需要予測
- 在庫残高
- 品目ポリシー
- 資金関連データ

### 5.2 必須機能

- 品目別需要予測を入力にして在庫リスクを計算できる
- 在庫リスクは品目カテゴリ別の閾値で判定できる
- MOQ と lot size を考慮した補充量候補を算出できる
- 将来の資金影響を評価できる
- シナリオ単位で比較できる
- 実行ごとに `run_id` を持ち、結果を再取得できる

### 5.3 出力

- `simulation_runs`
- `simulation_item_results`
- `simulation_report_v0.1` 系 JSON

### 5.4 非要求

- 顧客個票の保存や検索
- GUI 画面向けの一覧検索 API

## 6. GUI

### 6.1 全体導線

- `顧客`, `在庫`, `シミュレーション` の並列タブ導線を持てる
- GUI は薄いクライアントとして振る舞う
- GUI は API から受け取った情報を表示し、重い処理は内包しない

### 6.2 顧客画面

- 顧客一覧を表示できる
- 顧客詳細画面で購入履歴を見られる
- 顧客ごとの次回購入候補を表示できる
- 顧客ごとの候補は順位付きで表示できる

### 6.3 在庫画面

- 在庫一覧と在庫リスク一覧を表示できる
- 品目検索と絞り込みができる
- カテゴリ、在庫状態、リスク状態などで絞り込める

### 6.4 シミュレーション画面

- シミュレーションを起動できる
- 実行結果を見られる
- シミュレーションのレポート成果物を表示できる
- 実行中、完了、失敗などの状態を追跡できる

## 7. API

### 7.1 読み取り系

- 顧客、購入履歴、推薦結果、在庫、シミュレーション結果、レポート成果物を取得できる
- GUI が必要とする表示情報を API 経由で取得できる
- 画面が必要とする単位で情報を返せる

### 7.2 実行系

- ETL、分析、シミュレーションをジョブとして起動できる
- 長時間処理の状態を追跡できる
- ジョブ起動時に `job_id` または `run_id` を返せる

### 7.3 API 境界

- GUI は API だけを参照する
- GUI から DB、S3、ジョブ基盤へ直接アクセスしない

## 8. レポート成果物

- `decision-engine` は GUI と reporting が利用する JSON を出力できる
- reporting は JSON から図表とテキスト成果物を生成できる
- GUI は JSON または成果物参照を使って結果を表示できる

## 9. 機能間の整合条件

- 顧客前処理だけでは購入履歴機能は成立しない
- 購入傾向分析は `customers`, `orders`, `order_items`, `items` を前提にする
- `decision-engine` は顧客個票ではなく品目別需要予測のみを受ける
- GUI の機能は API 契約とレポート JSON 契約に依存する
