# プロダクト範囲

## 1. 文書の目的

この文書は、Decision Pack の対象範囲を固定するための仕様書である。実装、設計、画面、API、DB、AWS 配置の議論は、ここで定義する範囲を前提とする。

## 2. プロダクトの要約

Decision Pack は、顧客データ、購入履歴、商品品目、在庫、需要予測、意思決定支援を一体で扱う個人開発向け業務支援ツールである。
本プロジェクトは、既存の単体ツール群を共通の業務データ基盤と API 境界の上に再構成し、最終的には AWS 上で実行されるバックエンドと、薄いデスクトップ GUI の組み合わせへ移行することを目指す。

## 3. 想定利用者

### 3.1 主利用者

- 単独でデータ整備、在庫確認、購入傾向確認、意思決定支援を回したい個人開発者
- 自分の手元で業務データを把握しつつ、将来的にクラウド実行を行いたい利用者

### 3.2 利用者が行いたいこと

- 整形前の顧客データを、業務で使える構造へ直したい
- 顧客ごとの購入履歴を追跡したい
- 顧客ごとに次に買う可能性が高い商品を見たい
- 品目ごとの在庫状況と将来リスクを把握したい
- 補充判断を、需要予測と資金影響を踏まえて支援してほしい
- GUI から必要情報を引き出したいが、重い処理や永続化はエンジンに任せてローカルGUIは結果だけ受け取りたい。

## 4. 解決したい課題

### 4.1 現状課題

- 顧客データ前処理、意思決定計算、GUI が別々に存在し、責務境界が曖昧になりやすい
- 顧客CSVだけでは業務判断に必要な「誰が何を買ったか」が見えない
- 在庫一覧や顧客一覧の閲覧と、シミュレーション計算の責務が混ざりやすい
- 将来的に AWS へ載せたいが、ローカル GUI に処理が寄りすぎると移行が難しい
- 文書が薄いと、実装時に意味の揺れを判定しづらい。

### 4.2 本プロジェクトで解決すること

- 顧客データ、商品データ、購入履歴、在庫、予測、意思決定支援を一つの設計でつなぐ
- モジュールごとの責務を固定し、データの流れを明示する
- GUI は閲覧と操作に集中させ、重い処理は API とジョブに分離する
- 将来的な AWS 移行を前提に、ローカル段階から同じ境界で構築する

## 5. 提供価値

### 5.1 データ整備価値

- 顧客CSVと業務データを正規化し、再利用可能な業務データへ変換できる
- データ欠損や整形失敗を issue として残し、品質を追跡できる

### 5.2 業務可視化価値

- 顧客ごとの購入履歴と次回購入候補を一画面で確認できる
- 品目ごとの在庫状況とリスクを一覧で把握できる

### 5.3 意思決定支援価値

- 品目ごとの将来需要を見込んだ在庫評価ができる
- 欠品リスク、過剰在庫リスク、補充候補、資金影響をまとめて見られる
- シナリオ比較によって判断材料を増やせる

## 6. 今回の対象範囲

### 6.1 データ取り込み

- 顧客CSVの整形と PostgreSQL 永続化
- 商品品目リストの取り込み
- 受注と受注明細の取り込み
- 在庫残高または在庫元帳の取り込み

### 6.2 分析

- 顧客ごとの購入履歴集約
- 顧客ごとの次回購入候補 Top-N の算出
- 品目別需要予測の集約

### 6.3 意思決定支援

- 品目別需要予測を使った在庫リスク評価
- MOQ と lot size を考慮した補充提案
- 資金影響を含むシミュレーション

### 6.4 表示と操作

- 顧客一覧、顧客詳細、購入履歴、次回購入候補の表示
- 品目一覧、在庫一覧、在庫リスク一覧、検索と絞り込み
- シミュレーション実行、実行履歴、結果閲覧、レポート成果物表示

### 6.5 実行構造

- ローカルで成立する構成
- 将来 AWS にそのまま持っていける責務分割
- GUI を薄いクライアントに寄せる API 前提構造

## 7. 今回の非対象

### 7.1 業務連携

- POS、EC、ERP 等とのリアルタイム同期
- 外部 SaaS との双方向自動連携

### 7.2 業務自動化

- 自動発注の完全自律化
- 人手確認なしでの在庫政策変更

### 7.3 高度化の先送り

- 複雑な機械学習基盤の初期導入
- オンライン学習やリアルタイム推薦
- 高可用な分散構成の先行導入

### 7.4 組織利用前提の機能

- 複数テナント対応
- 厳密な権限管理の実装詳細
- 組織横断監査基盤との統合

## 8. スコープ境界の原則

- `decision-engine` は計算専用であり、顧客個票管理や一覧検索の責務は持たない
- 顧客個票は分析系までで扱い、`decision-engine` には `item_id` 単位の集約データだけを渡す
- GUI は DB や S3 を直接見ず、必ず `app-api` を介して情報を取得する
- レポート表示は `simulation_report_v0.1` 系 JSON 契約を境界にする

## 9. 段階的到達目標

### 9.1 第1段階

- 顧客、商品、受注、在庫の基本データを正規化して保存できる
- GUI で顧客一覧と在庫一覧が見られる

### 9.2 第2段階

- 購入傾向分析と次回購入候補が見られる
- GUI で購入傾向を確認できる

### 9.3 第3段階

- 需要予測を踏まえた在庫リスク評価と補充提案が見られる
- シミュレーション結果を GUI で閲覧できる

### 9.4 第4段階

- AWS 上の API とジョブ基盤でバックエンドを動かし、GUI を薄いラッパとして利用できる

## 10. この文書の前提

- 既存モジュールは全面的に再編されうる
- 文書が正本であり、コードはあとから追従する
- ローカルで成立させた後に AWS へ移す
- 本文は日本語を正本とする
