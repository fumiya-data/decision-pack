# 評価用サンプルデータ仕様

## 1. 文書の目的

この文書は、Decision Pack のローカル評価と推薦評価に使う標準サンプルデータの規模、分布、生成規則を定義する。
ここで定義する仕様は、`commerce-etl` のサンプル生成機能と `purchase-insights` のオフライン評価の前提となる。

## 2. 標準サンプルの規模

標準サンプルは次の件数を正本とする。

- 顧客: `50,000`
- 商品: `100`
- 注文: `150,000`

これに加えて、注文 1 件あたりの明細数は平均 `2.6` から `3.0` を狙い、受注明細は概ね `390,000` から `450,000` 件の範囲に収まることを期待する。

## 3. サンプルの目的

このサンプルは、次の 3 つを同時に成立させるためのものである。

- 顧客前処理、取込、DB 永続化、API、GUI の負荷確認
- `purchase-insights` の推薦評価
- `decision-engine` の在庫・補充判断の入力確認

したがって、単に件数だけ多いデータではなく、繰り返し購入、カテゴリ偏り、人気の偏り、セグメント差が見えることを要件とする。

## 4. 顧客データ前提

- 顧客元データは未整形の raw CSV を起点とする
- `customers-etl` を通した正規化済み顧客 CSV をサンプル生成入力に使う
- 顧客件数は `50,000` を前提とする
- 顧客には `country`, `status`, `tier`, `preferred_language` が含まれることを期待する
- raw 顧客データには英語、日本語、ヒンディー語、中国語の氏名を含める
- `preferred_language` は少なくとも `en-US`, `ja`, `hi`, `zh-CN` を表現できること
- 氏名、住所、都市、地域、郵便番号、電話番号は国ごとの自然な表現に対応していること

## 5. 商品データ仕様

### 5.1 件数

- 商品件数は `100`

### 5.2 カテゴリ

- カテゴリ件数は `10`
- 1 カテゴリあたりの代表商品数は概ね `10`
- 例:
  - `apparel`
  - `beauty`
  - `electronics`
  - `food`
  - `home`
  - `office`
  - `outdoor`
  - `pet`
  - `sports`
  - `wellness`

### 5.3 商品属性

- `item_id`
- `item_name`
- `category`
- `uom`
- `is_active`
- `lead_time_days`
- `moq`
- `lot_size`

## 6. 注文データ仕様

### 6.1 件数

- 注文件数は `150,000`

### 6.2 繰り返し購入条件

- 評価対象顧客の十分な母数を作るため、`2回以上購入` の顧客を十分に含める
- 最低条件として、全顧客の `60%` 以上が `2回以上購入` を持つこと
- 推奨目標として、`2回以上購入` 顧客は `30,000` 件以上とする

### 6.3 期間

- 注文日時は約 `24か月` の範囲に分布させる
- 時系列が崩れないこと
- 顧客ごとの注文列に前後関係があること

### 6.4 注文ステータス

- 注文状態は `shipped` または `completed` を中心とする
- 未確定注文ばかりになる構成は避ける

## 7. 受注明細データ仕様

### 7.1 行数

- 受注明細数は `390,000` から `450,000` 件程度を目安とする

### 7.2 1 注文あたりの明細数

- 最小 `1`
- 最大 `5`
- 平均 `2.6` から `3.0`

### 7.3 数量

- 数量は `1` から `4` を中心とする
- 大半は `1` または `2`

## 8. 分布要件

### 8.1 人気の偏り

- 商品人気は完全一様ではなく、ロングテールを持つこと
- 上位人気商品だけが過度に支配しないこと

### 8.2 セグメント差

- `country`, `status`, `tier` によってカテゴリ選好が変化すること
- 同一セグメント内である程度似た購買傾向が見えること
- 顧客 raw データの `country` と地理属性の間に矛盾がないこと
- 英語圏名で日本住所のような不自然な組み合わせを標準サンプルでは避けること

### 8.3 遷移パターン

- 同じカテゴリを連続購入する傾向を一定程度含むこと
- 前回購入カテゴリと次回購入カテゴリの相関が全くゼロにならないこと

### 8.4 再購入パターン

- 同一顧客が同一商品を再購入するケースを十分に含むこと
- 再購入だけで全て説明できるデータにはしないこと

## 9. 在庫サンプル仕様

- 在庫は全 `100` 商品に対して生成する
- `inventory_balance` は `on_hand`, `on_order`, `reserved_qty` を持つ
- `on_hand` は 0 ばかりにならないこと
- `on_order` は全商品ゼロではなく、リードタイム付き補充の入力として利用可能であること

## 10. 生成成果物

`commerce-etl --generate-sample` は次の CSV を出力する。

- `items.csv`
- `orders.csv`
- `order_items.csv`
- `inventory.csv`
- `sample_metadata.json`

## 11. 期待する評価成立条件

このサンプルは、次の評価が成立することを目的とする。

- `Hit@3`
- `Hit@5`
- `Recall@5`
- `Recall@10`
- `NDCG@5`
- `NDCG@10`
- 人気順ベースラインとの差分確認

## 12. 運用方針

- 巨大サンプル CSV は原則として Git 管理しない
- 生成規則と seed を文書と metadata に残し、必要時に再生成する
- 実験 seed を変える場合でも、標準 seed は固定して比較軸を維持する

### 12.1 full サンプルの扱い

- `50,000顧客 / 100商品 / 150,000注文` の full サンプルは、開発側のローカル検証専用とする
- full サンプルの raw CSV、整形済み CSV、受注 CSV、評価結果 CSV は Git と GitHub に反映しない
- リポジトリに残す正本は、生成器、seed、仕様書、metadata、smoke 用または dev 用の小中規模サンプルとする
- full サンプルが必要な場合は、各開発環境で再生成する
