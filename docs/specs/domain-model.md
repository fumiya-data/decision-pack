# ドメインモデル

## 1. 文書の目的

この文書は、Decision Pack で扱う主要ドメイン、エンティティ、識別子、境界、不変条件を定義する。
ここで定義した意味は、DB 設計、API 契約、`decision-engine` の入力境界、Lean での形式仕様の基礎になる。

## 2. 主要ドメイン

### 2.1 顧客ドメイン

顧客マスタ、顧客属性、顧客整形時の問題を扱う。
主な責務は、顧客を業務利用可能な構造へ変換し、品質状態を追跡可能にすることである。

### 2.2 商品・販売ドメイン

商品品目、注文、注文明細を扱う。
主な責務は、顧客がどの商品をいつどの数量で購入したかを再構成できるようにすることである。

### 2.3 在庫ドメイン

在庫残高、在庫移動、補充制約、在庫政策を扱う。
主な責務は、ある時点の品目在庫と、補充判断に必要な制約を保持することである。

### 2.4 購入傾向分析ドメイン

顧客別の次回購入候補と、品目別需要予測を扱う。
主な責務は、顧客視点の分析結果と、意思決定支援用に集約された需要情報を分離して作ることである。

### 2.5 意思決定支援ドメイン

シミュレーション実行、品目別結果、レポート成果を扱う。
主な責務は、需要予測、在庫、資金制約を入力にして、判断材料となる結果を生成することである。

## 3. 主要エンティティ

### 3.1 顧客ドメイン

#### `customer`

- 業務利用可能な顧客レコード
- 連絡先、属性、状態、購入関連属性を保持する
- 顧客個票として扱う

#### `customer_load_issue`

- 顧客データ取込時に検出した問題
- 欠損、解釈不能、規格外値、補正不能値を保持する
- データ品質追跡のために残す

### 3.2 商品・販売ドメイン

#### `item`

- 管理対象となる商品品目
- 需要予測、在庫、補充提案の基準単位

#### `order`

- 顧客による注文のヘッダ
- 注文日時、顧客、金額、状態などの単位

#### `order_item`

- 注文明細
- `order` と `item` を結びつける
- 数量、単価、小計などを保持する

### 3.3 在庫ドメイン

#### `inventory_balance`

- ある時点の品目別在庫残高
- `decision-engine` が直接使う主要入力の一つ

#### `inventory_movement`

- 在庫増減の履歴
- 必須ではないが、監査や再構築に有用

#### `item_policy`

- 補充判断に必要なポリシー
- 例: 安全在庫、リードタイム、MOQ、lot size、カテゴリ別閾値

### 3.4 購入傾向分析ドメイン

#### `customer_item_next_buy_score`

- 顧客別の次回購入候補
- `customer_id` と `item_id` の組に対して `score` と `rank` を持つ
- GUI が顧客別候補を表示するために使う

#### `item_demand_forecast`

- `decision-engine` へ渡す品目別需要予測
- 顧客個票を含まず、品目単位に集約される

### 3.5 意思決定支援ドメイン

#### `simulation_run`

- シミュレーション実行の単位
- 入力条件、実行日時、状態、識別子を持つ

#### `simulation_item_result`

- シミュレーション結果の品目単位明細
- 在庫リスク、補充候補、需要、欠品量などを含みうる

#### `simulation_report`

- GUI と reporting が参照するレポート成果
- `simulation_report_v0.1` 系 JSON を中心とする

## 4. 集約と境界

### 4.1 顧客個票の境界

- 顧客個票は `customers-etl`、`commerce-etl`、`purchase-insights` までは扱う
- `decision-engine` には顧客個票を持ち込まない
- `decision-engine` が受けるのは `item_id` 単位の需要予測と在庫情報である

### 4.2 購入傾向と意思決定の境界

- `customer_item_next_buy_score` は顧客向け表示用の分析成果である
- `item_demand_forecast` は意思決定支援向けの集約成果である
- 両者は同じ分析過程から生まれても、同じ意味のデータではない

### 4.3 GUI と計算の境界

- GUI は一覧、詳細、実行状態、結果表示の責務を持つ
- 意思決定計算や DB 参照の責務は持たない
- GUI は API 契約とレポート JSON 契約だけを見る

## 5. 識別子の方針

### 5.1 共通方針

- DB と API の識別子は `snake_case` の英語名を使う
- 識別子の意味は文書と SQL で固定する

### 5.2 顧客系

- `customer_id` は顧客個票の識別子として業務DBと分析系で使う
- `decision-engine` 内では `customer_id` を主要識別子として扱わない

### 5.3 商品・意思決定系

- `item_id` は商品、在庫、需要予測、補充提案を横断する主要識別子とする
- `scenario_id` はシナリオの識別子とする
- `run_id` は実行単位の識別子とする

## 6. 不変条件

- `order_item` は必ず `order` と `item` に紐づく
- `item_demand_forecast` は顧客個票を含まない
- `simulation_item_result` は必ず対応する `simulation_run` に属する
- `decision-engine` は `customer_item_next_buy_score` を直接入力にしない
- MOQ と lot size は補充制約であり、需要予測そのものではない

## 7. 境界づけられた責務の意味

- 顧客整形の誤りは `customers-etl` で扱う
- 商品、受注、在庫の統合は `commerce-etl` で扱う
- 顧客ごとの次回購入候補と品目別需要予測の生成は `purchase-insights` で扱う
- 補充提案、在庫リスク、資金影響は `decision-engine` で扱う
- 表示と操作は `desktop-ui`、境界は `app-api` で扱う
