# 既存モジュール移行計画書

## 1. 目的

本計画書は、現在の `customers_etl`、`engine`、`ui`、`reporting`、`spec`、`data` を、今後採用する責務分割とディレクトリ構造へ移すための移行方針を定めるものです。

本計画書の対象はコード移設そのものではなく、次を固定することです。

- どの既存モジュールを残すか
- どの責務を別モジュールへ切り出すか
- どの順番で移行するか
- どの時点で旧構造を廃止するか

## 2. 前提

- 既存モジュールは、現行構造のまま温存する前提ではない
- 新しい責務分割に合わない箇所は、破壊的に再編してよい
- 先に文書と構造を固定し、その後にコードを移す
- 一時的な互換レイヤは許容するが、恒久化は行わない

## 3. 目標のディレクトリ構造

```text
decision-pack/
  docs/
  customers-etl/
  commerce-etl/
  purchase-insights/
  decision-engine/
  app-api/
  desktop-ui/
  reporting/
  spec/
  data/
```

## 4. 既存モジュールの移行方針

### 4.1 `customers_etl` -> `customers-etl`

#### 据え置き

- 顧客CSVの復旧と正規化ロジック
- issue 集計
- CLI 実行の基本骨格

#### 見直し

- `engine` 向けを前提にした下流成果物の位置づけ
- ローカルファイル中心の前提
- PostgreSQL 永続化の欠如

#### 責務変更

- 顧客マスタの正規化
- `customers` と `customer_load_issues` の永続化
- 必要に応じた `run_summary` の記録

#### 他モジュールへ渡す責務

- 品目、受注、在庫の取込は `commerce-etl` へ渡す
- 購入傾向分析は `purchase-insights` へ渡す
- 意思決定計算入力の生成は `decision-engine` 側の契約へ合わせて再定義する

### 4.2 `engine` -> `decision-engine`

#### 据え置き

- `domain`
- `sim`
- `report`
- シミュレーションの中核ロジック

#### 見直し

- 現行の入力前提
- ローカル quickstart バイナリ前提
- 顧客と購入履歴を将来ここへ入れる可能性

#### 責務変更

- `item_demand_forecast`
- `inventory_balance`
- `item_policy`
- `cashflow` 系データ

を受けて、在庫・資金の意思決定計算を行うことに責務を絞る。

#### 持たないこととする責務

- 顧客一覧
- 購入履歴検索
- 在庫一覧の CRUD
- API の読み取り系クエリ

### 4.3 `ui` -> `desktop-ui`

#### 据え置き

- Iced ベースのデスクトップ UI
- 画面遷移と状態表示の骨格

#### 見直し

- ローカル JSON ファイル読込中心の前提
- `reporting` CLI をローカルで直接呼ぶ前提

#### 新しい責務

- 顧客一覧、購入履歴、購入傾向、在庫一覧、シミュレーション結果の表示
- `app-api` 経由でのデータ取得
- ジョブ起動後の状態表示

#### 持たないこととする責務

- DB 接続
- S3 参照
- ETL 実行本体
- シミュレーション本体

### 4.4 `reporting` -> `reporting`

#### 据え置き

- `simulation_report_v0.1` JSON から図表とテキストを生成する役割
- Python による表現レイヤ

#### 見直し

- ローカル出力専用の前提
- 旧来のサンプルや補助文書の位置づけ

#### 新しい責務

- `decision-engine` が出したレポート JSON の可視化
- API またはジョブ成果物としての図表生成

#### 持たないこととする責務

- 意思決定計算
- GUI 表示
- DB 参照

### 4.5 `spec` -> `spec`

#### 据え置き

- 形式仕様の正本としての役割

#### 見直し

- 旧ディレクトリ名や旧責務分割を前提にした表現

#### 新しい責務

- `decision-engine` の制約、不変条件、計算意味の記述
- 必要に応じて、データ契約の論理条件の記述

### 4.6 `data` -> `data`

#### 据え置き

- fixture
- raw sample
- expected output

#### 見直し

- 顧客CSV偏重の構成
- 商品、受注、在庫 fixture の不足

#### 新しい責務

- モジュール別 fixture の整理
- 顧客、商品、受注、在庫、需要予測、レポート JSON のサンプル保管

## 5. 新設するモジュール

### `commerce-etl`

- 商品品目、受注、受注明細、在庫取込を担当する
- 既存モジュールからの流用候補は限定的で、新規作成を前提とする

### `purchase-insights`

- モジュールA（仮名）の正式名称
- 顧客ごとの次回購入候補推定と品目需要予測を担当する
- 既存モジュールからの直接流用はほぼ想定しない

### `app-api`

- GUI のための統一 API 境界
- 読み取り系 API とジョブ起動 API を提供する
- 既存モジュールからの直接流用はほぼ想定しない

## 6. 移行の進め方

### フェーズ 1: 契約を固定

- PostgreSQL の論理スキーマを定義する
- API 契約を定義する
- `decision-engine` 入力契約を定義する

### フェーズ 2: 新ディレクトリの受け皿を作成

- `customers-etl`
- `commerce-etl`
- `purchase-insights`
- `decision-engine`
- `app-api`
- `desktop-ui`

の最終ディレクトリを決定する。

この段階では、旧ディレクトリをすぐ消さずに、受け皿を先に作る。

### フェーズ 3: 既存コードを再配置

- `customers_etl` の中核を `customers-etl` へ移す
- `engine` の中核を `decision-engine` へ移す
- `ui` の中核を `desktop-ui` へ移す
- `reporting` は必要に応じて整理のみ行う

### フェーズ 4: 新設モジュールを実装

- `commerce-etl`
- `purchase-insights`
- `app-api`

を追加する。

### フェーズ 5: 旧構造を段階的に廃止

- 新構造で主要機能が成立したら、旧ディレクトリを段階的に廃止する
- 廃止前に必要な compatibility wrapper が残っていないことを確認する

## 7. 互換期間のルール

- 旧ディレクトリから新ディレクトリへの委譲は一時的に許容する
- 互換期間中に新しい責務を旧構造へ追加しない
- 互換レイヤは最終的に削除する前提で書く

## 8. 移行順の優先度

1. `customers_etl` -> `customers-etl`
2. `engine` -> `decision-engine`
3. `ui` -> `desktop-ui`
4. `commerce-etl` 新設
5. `purchase-insights` 新設
6. `app-api` 新設
7. `reporting` と `spec` の微調整

この順にする理由は、既存コードの再配置を先に完了させる方が、新設モジュールとの境界を明確に保ちやすいため。

## 9. 予想されるリスク

### リスク 1

旧モジュール名のまま議論や実装を続けたために、文書とコードの呼び名がずれる

### 対策

- 文書では新しいモジュール名を正本にする
- 実装時も、新ディレクトリ名を先に切ることを規約とする

### リスク 2

`decision-engine` に顧客個票や検索責務が流入する

### 対策

- API と DB の読み取り責務を `app-api` に固定する
- 顧客分析は `purchase-insights` に固定する

### リスク 3

旧構造の互換レイヤが恒久化する

### 対策

- 互換レイヤを置くなら、削除条件を先に定める

## 10. 完了条件

- 新しいディレクトリ構造で各モジュールの責務が説明できる
- 既存モジュールの移行先がすべて定義されている
- 新設モジュールの責務が既存モジュールと重複していない
- 旧構造をいつ廃止するかが定められている
