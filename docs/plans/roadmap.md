# ロードマップ

## フェーズ 0: 文書と構造の固定

### 目的

実装前に、命名規則、文書体系、責務分割、ディレクトリ構造を固定する。

### 完了条件

- `docs/` 配下の正本文書が揃っている
- ディレクトリ構造案が確定している
- ADRとして実際に記録されている

## フェーズ 1: データ基盤の再設計

### 目的

顧客、商品、受注、在庫の正規化されたデータモデルを作る。

### 完了条件

- PostgreSQL の最小スキーマが定義されている
- `customers-etl` と `commerce-etl` の入力契約が定義されている
- 移行用 fixture とサンプルが定義されている

## フェーズ 2: 分析モジュールの追加

### 目的

購入履歴から次回購入候補と品目別需要予測を作る。

### 完了条件

- `purchase-insights` の MVP アルゴリズムが実装できる仕様になっている
- `customer_item_next_buy_score` と `item_demand_forecast` の契約が定義されている

## フェーズ 3: 意思決定支援エンジンの再定義

### 目的

需要予測、在庫、資金を統合した意思決定計算を成立させる。

### 完了条件

- `decision-engine` の入力契約と出力契約が固定されている
- レポート JSON の拡張方針が明確になっている

## フェーズ 4: API と GUI の再構成

### 目的

GUI を API 前提の薄いクライアントへ寄せる。

### 完了条件

- `app-api` の主要エンドポイントが固定されている
- 顧客、購入履歴、在庫、シミュレーションの画面責務が決まっている

## フェーズ 5: AWS 化

### 目的

ローカル実装を AWS 上の API とジョブ実行基盤へ移す。

### 完了条件

- S3、RDS、API Gateway、Lambda、Step Functions、ECS/Fargate の責務が固定されている
- GUI が AWS の内部サービスを直接参照しない
