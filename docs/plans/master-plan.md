# Decision Pack 全体計画書

## 1. 目的

Decision Pack を、顧客データ、購入履歴、在庫、資金、需要予測、意思決定支援を一貫して扱える個人開発向けプロダクトとして再設計する。

## 2. 到達点

- 顧客CSVを整形し、PostgreSQLへ永続化できる
- 商品品目、購入履歴、在庫を一貫したデータモデルで持てる
- 顧客ごとの次回購入候補を推定できる
- 推定需要を用いて在庫リスクと補充提案を計算できる
- GUI で顧客、購入履歴、購入傾向、在庫状況、シミュレーション結果、レポート成果物を見ることができる
- 将来的には AWS 上の API とジョブ基盤へ移行できる

## 3. 基本方針

- 既存モジュールはそのまま流用せず、責務に合わせて再配置する( [cf. 既存モジュール移行計画](existing-module-migration-plan.md) )
- `engine` は意思決定計算専用とし、一覧検索や顧客管理を持たせない
- GUI は薄いクライアントに寄せて、差し替えやすくする
- 個票データと計算入力を分離し、`engine` には集約済み需要データのみを渡す
- 先に仕様と構造を詳細に固めておき、実装はその後に着手する。

## 4. 主要モジュール

- `customers-etl`
- `commerce-etl`
- `purchase-insights`
- `decision-engine`
- `app-api`
- `desktop-ui`
- `reporting`
- `spec`

## 5. 開発の進め方

### STEP 1

- 命名規則と文書体系を固定する
- 最終ディレクトリ構造を決める
- 仕様、設計、ADR、進捗、バックログの文書を作る

### STEP 2

- PostgreSQL スキーマとデータ契約を固める
- `customers-etl` と `commerce-etl` の役割を定める
- `purchase-insights` の MVP アルゴリズムを定める

### STEP 3

- `decision-engine` の入力契約と出力契約を再定義する
- `app-api` と `desktop-ui` の境界を定める

### STEP 4

- まずはローカルで全体を一通り成立させる
- その後AWS 実行構成へ持ち上げる

## 6. 優先順位

1. 文書体系
2. ディレクトリ構造
3. データモデル
4. API 境界
5. ローカル実装
6. AWS 化

## 7. 完了条件

- 文書のみをもって全体構成と責務が説明可能であること
- ディレクトリ構造と命名規則が固定され、遵守されていること
- モジュール間の入力と出力が明文化されていること
- 実装前に検討すべき論点が ADR またはバックログに落とし込んでいること
